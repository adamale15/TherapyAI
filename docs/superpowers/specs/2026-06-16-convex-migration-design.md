# Convex Migration Design

## Goal

Move Vesh from Supabase-backed application data to Convex as the long-term backend while keeping Clerk as the only user authentication system.

## Chosen Approach

Use Convex directly for product data and keep Next.js route handlers only for server-only operations such as Gemini chat replies, ElevenLabs proxying, and file upload parsing. This creates a clean long-term split:

- Clerk owns identity and route protection.
- Convex owns personas, chat sessions, and completed session history.
- Next.js API routes own secret-bearing integration work.
- React components read and mutate app data through Convex where possible.

This is better long term than a compatibility wrapper because it removes the current Clerk ID versus Supabase UUID mismatch and avoids preserving old custom password auth code.

## Architecture

Convex will contain the app data model in `convex/schema.ts` and focused function modules:

- `convex/personas.ts` for default persona loading, user persona loading, saving, and deletion.
- `convex/chatSessions.ts` for session creation, persona selection, message history, reset, and completed history.
- `convex/seed.ts` for idempotent default persona seeding from structured constants.
- `convex/auth.config.ts` for Clerk token validation.

The Next.js app will provide a `ConvexProviderWithClerk` client provider in the root layout. Client components can use Convex React hooks for persona management. Route handlers that must run on the server will use the Convex HTTP client.

## Data Model

Convex tables:

- `personas`: stores default and custom personas. `ownerClerkId` is optional for default personas and set for custom personas.
- `chatSessions`: stores active session state keyed by a `sessionKey` generated by the client.
- `sessionHistory`: stores completed session summaries and transcripts.

All user-owned data uses Clerk user IDs directly. No UUID conversion layer is needed.

## Data Flow

Persona selection and management:

1. The React UI queries Convex for default personas plus the current user's custom personas.
2. Custom persona uploads still go through Next.js because document parsing and Gemini generation are server-only.
3. The upload route saves the generated persona through a Convex mutation.

Chat:

1. The client sends user text to `/api/chat` with the session key.
2. The route loads the current session from Convex.
3. The route calls Gemini with the session history and persona.
4. The route appends the user and assistant turns back into Convex.
5. The client receives the reply and updates local session UI.

This keeps Gemini credentials out of the browser while allowing Convex to own persistent session state.

## Cleanup Scope

Remove Supabase-specific runtime code and active API paths:

- Delete or retire Supabase clients and auth services.
- Remove `/api/auth/login` and `/api/auth/register` from the active app.
- Remove Supabase dependencies and docs that describe the old active setup.
- Rename Supabase-oriented services to Convex-oriented equivalents.

Polish while touching code:

- Remove hardcoded Gemini API key fallback.
- Stop treating ElevenLabs server credentials as public client configuration.
- Centralize browser API URL construction instead of duplicating it in components.
- Keep UI behavior stable unless a split is needed to make the Convex boundary clean.

## Error Handling

If Convex is not configured locally, the app should fail with a clear environment error instead of silently using stale Supabase fallback behavior. If default personas are not seeded, the UI can still use bundled fallback constants for first-run development, but saved Convex data is the long-term source of truth.

## Testing

Verification should include:

- Unit tests for persona data mapping and fallback behavior.
- Unit tests for chat session storage helpers where possible.
- Type checking with `npm run type-check`.
- Build verification with `npm run build` if Convex code generation and required environment variables are available.

## Open Operational Requirement

Running the migrated app requires a Convex deployment and generated Convex client files. Local setup is expected to use:

```bash
npm install
npx convex dev
```

The Convex dashboard must receive Clerk issuer configuration through `CLERK_JWT_ISSUER_DOMAIN`, and Next.js must have `NEXT_PUBLIC_CONVEX_URL`.
